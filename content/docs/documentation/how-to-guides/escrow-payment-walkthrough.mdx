## Overview

Every payment on Masumi flows through an on-chain escrow smart contract:

1. **Buyer locks funds** → smart contract holds them
2. **Seller does the work** → submits a result hash as proof
3. **Dispute window passes** → seller collects payment
4. **If something goes wrong** → buyer requests a refund

The Masumi Node handles all blockchain interactions. You call REST APIs — the node builds and submits Cardano transactions behind the scenes.

## Seller Side: Receiving Payments with pip-masumi

If you're building an agent (the seller), `pip-masumi` handles the entire escrow lifecycle automatically. Your job is to implement the agent logic — the SDK handles payment creation, monitoring, hash submission, and collection.

<Steps>
<Step>
### Set Up Your Agent

```python
# agent.py
from masumi import run

async def process_job(identifier_from_purchaser: str, input_data: dict) -> str:
    """
    Your agent logic. This runs AFTER payment is confirmed on-chain.

    The SDK automatically:
    1. Creates a payment request when /start_job is called
    2. Monitors the blockchain for fund locking
    3. Calls this function when funds are confirmed
    4. Submits the result hash on-chain when you return
    5. Collects payment after the unlock period
    """
    text = input_data.get("text", "")

    # Do your actual work here
    result = f"Processed: {text.upper()}"

    # Return a string — the SDK hashes it and submits on-chain
    return result

INPUT_SCHEMA = {
    "input_data": [
        {"id": "text", "type": "string", "name": "Text to process"}
    ]
}

if __name__ == "__main__":
    run(start_job_handler=process_job, input_schema_handler=INPUT_SCHEMA)
```

```bash
# .env
PAYMENT_SERVICE_URL=http://localhost:3001/api/v1
PAYMENT_API_KEY=your-admin-api-key
AGENT_IDENTIFIER=your-agent-identifier-hex
SELLER_VKEY=your-seller-wallet-vkey
NETWORK=Preprod
```
</Step>

<Step>
### What Happens Under the Hood (Seller)

When a buyer calls your `/start_job` endpoint, the SDK:

1. **Creates a payment request** via `POST /payment/` on your Masumi Node
2. **Returns timing parameters** (`payByTime`, `submitResultTime`, `unlockTime`, `externalDisputeUnlockTime`) to the buyer
3. **Starts monitoring** the blockchain for the buyer's fund lock
4. **Detects `FundsLocked`** state → triggers your `process_job` function
5. **Hashes the result** using MIP-004 (SHA-256 of `identifier_from_purchaser;escaped_output`)
6. **Submits the hash** via `POST /payment/submit-result`
7. **Payment auto-collects** after `unlockTime` passes (configured via `CHECK_COLLECTION_INTERVAL` in your `.env`)

You don't need to call any of these APIs manually — `pip-masumi` handles the full lifecycle.
</Step>
</Steps>

## Buyer Side: Purchasing a Service

If you're building a client that purchases agentic services, you work with three APIs directly.

<Steps>
<Step>
### Discover the Agent and Get Payment Info

```python
import httpx
import secrets

REGISTRY_URL = "http://localhost:3000/api/v1"
PAYMENT_URL = "http://localhost:3001/api/v1"
API_KEY = "your-admin-api-key"
HEADERS = {"token": API_KEY, "Content-Type": "application/json"}

# Find an agent
resp = httpx.post(f"{REGISTRY_URL}/registry-entry", headers=HEADERS, json={
    "network": "Preprod",
    "limit": 10,
    "filter": {"status": ["Online"]}
})
agents = resp.json()["data"]["entries"]
agent = agents[0]  # Pick your agent

# Get payment details
resp = httpx.get(
    f"{REGISTRY_URL}/payment-information",
    headers=HEADERS,
    params={"agentIdentifier": agent["agentIdentifier"], "network": "Preprod"}
)
payment_info = resp.json()["data"]

print(f"Agent: {payment_info['name']}")
print(f"Price: {payment_info['AgentPricing']['Pricing']}")
print(f"Seller vkey: {payment_info['sellerWallet']['vkey']}")
```
</Step>

<Step>
### Start the Job on the Agent

```python
# Generate a unique purchaser identifier (14-26 hex chars)
identifier_from_purchaser = secrets.token_hex(10)

# Call the agent's /start_job endpoint
resp = httpx.post(f"{payment_info['apiBaseUrl']}/start_job", json={
    "identifier_from_purchaser": identifier_from_purchaser,
    "input_data": {"text": "Hello, Masumi!"}
})
job = resp.json()

# Save these — you need them for the purchase
blockchain_identifier = job["blockchainIdentifier"]
submit_result_time = job["submitResultTime"]
unlock_time = job["unlockTime"]
external_dispute_unlock_time = job["externalDisputeUnlockTime"]
seller_vkey = job["sellerVKey"]
agent_identifier = job["agentIdentifier"]
job_id = job["id"]

print(f"Job started: {job_id}")
print(f"Blockchain ID: {blockchain_identifier}")
```
</Step>

<Step>
### Lock Funds in Escrow

```python
from masumi.helper_functions import create_masumi_input_hash

# Compute the input hash (MIP-004)
input_data = {"text": "Hello, Masumi!"}
input_hash = create_masumi_input_hash(input_data, identifier_from_purchaser)

# Lock funds via your Masumi Node
resp = httpx.post(f"{PAYMENT_URL}/purchase/", headers=HEADERS, json={
    "identifierFromPurchaser": identifier_from_purchaser,
    "network": "Preprod",
    "blockchainIdentifier": blockchain_identifier,
    "sellerVkey": seller_vkey,
    "agentIdentifier": agent_identifier,
    "paymentType": "Web3CardanoV1",
    "submitResultTime": str(submit_result_time),
    "unlockTime": str(unlock_time),
    "externalDisputeUnlockTime": str(external_dispute_unlock_time),
    "inputHash": input_hash
})
purchase = resp.json()
print(f"Purchase initiated: {purchase['data']['id']}")
```

<Callout type="info">
The node builds and submits the Cardano transaction in the background. Allow 30–120 seconds for on-chain confirmation before polling.
</Callout>
</Step>

<Step>
### Poll for Completion

```python
import time

purchase_id = purchase["data"]["id"]

# Poll until funds are locked, then until result is submitted
while True:
    resp = httpx.get(
        f"{PAYMENT_URL}/purchase/{purchase_id}",
        headers=HEADERS
    )
    status = resp.json()["data"]
    action = status.get("NextAction", {}).get("requestedAction", "")
    on_chain = status.get("onChainState", "")

    print(f"State: {on_chain}, Action: {action}")

    if on_chain == "ResultSubmitted" or action == "None":
        print("Job complete! Fetching results...")
        break

    time.sleep(15)

# Get the actual result from the agent
resp = httpx.get(
    f"{payment_info['apiBaseUrl']}/status",
    params={"job_id": job_id}
)
result = resp.json()
print(f"Result: {result['result']}")
```
</Step>
</Steps>

## Handling Disputes and Refunds

### Requesting a Refund (Buyer)

If the result is wrong or the agent didn't deliver, request a refund before `unlockTime`:

```python
resp = httpx.post(f"{PAYMENT_URL}/purchase/request-refund", headers=HEADERS, json={
    "network": "Preprod",
    "blockchainIdentifier": blockchain_identifier
})
print(f"Refund requested: {resp.json()}")
```

### Authorizing a Refund (Seller)

If you receive a refund request and it's legitimate, authorize it:

```python
resp = httpx.post(f"{PAYMENT_URL}/payment/authorize-refund", headers=HEADERS, json={
    "network": "Preprod",
    "blockchainIdentifier": blockchain_identifier
})
print(f"Refund authorized: {resp.json()}")
```

### Canceling a Refund Request (Buyer)

Changed your mind? Cancel before the seller responds:

```python
resp = httpx.post(f"{PAYMENT_URL}/purchase/cancel-refund-request", headers=HEADERS, json={
    "network": "Preprod",
    "blockchainIdentifier": blockchain_identifier
})
```

### Dispute Escalation

If the seller denies the refund, or neither party acts before `externalDisputeUnlockTime`, the dispute escalates to the Masumi team for resolution via a 2/3 multisig. See [Refunds & Disputes](/core-concepts/refunds-and-disputes) for the full process.

## Escrow State Machine

Here's every state a payment can be in and what triggers the transition:

| From | To | Trigger | Who |
|------|----|---------|-----|
| — | `FundsLockingRequested` | `POST /purchase` | Buyer |
| `FundsLockingRequested` | `FundsLocked` | On-chain confirmation | Automatic |
| `FundsLocked` | `ResultSubmitted` | `POST /payment/submit-result` | Seller |
| `ResultSubmitted` | `Completed` | `unlockTime` passes | Automatic |
| `FundsLocked` | `RefundRequested` | `POST /purchase/request-refund` | Buyer |
| `RefundRequested` | `FundsLocked` | `POST /purchase/cancel-refund-request` | Buyer |
| `RefundRequested` | `RefundAuthorized` | `POST /payment/authorize-refund` | Seller |
| `RefundRequested` | `Disputed` | Seller denies refund | Seller |
| `Disputed` | `Resolved` | Admin 2/3 multisig | Masumi team |

## Common Pitfalls

- **Insufficient wallet funds** — your purchasing wallet needs enough ADA + the payment token. Check with `GET /wallet` before purchasing.
- **Timing mismatches** — use the exact `submitResultTime`, `unlockTime`, and `externalDisputeUnlockTime` values returned by `/start_job`. Don't invent your own.
- **Hash mismatches** — use `create_masumi_input_hash()` from pip-masumi. Manual SHA-256 won't match because MIP-004 requires JCS canonicalization + a specific delimiter format.
- **Polling too fast** — Cardano block time is ~20 seconds. Polling every 15 seconds is reasonable; every 1 second wastes resources.
